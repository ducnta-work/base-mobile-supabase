# Engineering Principles

This document captures the recurring architectural and coding conventions. Use it as reusable rules for new projects and for AI agent guidance.

## 1. Project Layout & Entry
- Entry point is `index.ts`, which bootstraps `src/app`.
- Core folders:
  - `src/app`: top-level app wiring and providers.
  - `src/screens/*`: feature screens (each feature uses a folder with `index.tsx` + `Layout.tsx`).
  - `src/components`: shared UI components.
  - `src/navigation`: navigation stacks, tabs, drawers, and helpers.
  - `src/styles`: theme and styling system setup.
  - `src/client`: app-level client wiring and React Query config.
  - `src/localization`: i18n resources and helpers.
  - `src/assets`: static assets with auto-generated registry.
- Local workspaces live in `modules/` (e.g., Supabase client + repos, internal UI kit).

## 2. Screens: Container vs Layout Split
- Each screen folder uses:
  - `index.tsx`: data fetching, state, navigation, mutations, effects.
  - `Layout.tsx`: pure UI and layout rendering with props.
- Layout components are stateless where possible; all side effects stay in the container.
- Shared UI (loading/error) uses overlay components instead of inline spinners.

## 3. Data Fetching & State (React Query)
- React Query is the standard for server state.
- Query definitions live in `src/client/queries.ts` using `queryOptions` and consistent `queryKey`s.
- Containers call `useQuery` or `useMutation`, then pass data into Layout.
- After successful mutations, invalidate relevant queries via `queryClient.invalidateQueries`.
- Avoid async logic in Layout components; keep it in container.

## 4. Supabase Client & Repository Pattern
- Supabase access is centralized in a workspace module (`modules/family-task-client`).
- App-level client wiring is in `src/client/index.ts`:
  - Create client once, export repos (`userRepo`, `taskRepo`, etc.).
  - Repos are created by factory functions like `createTaskRepository(client)`.
- Repository rules:
  - Functions return typed data from `database/model`.
  - Always throw Supabase errors instead of swallowing them.
  - Use helper utilities for shared logic (e.g., `getUser`, recurrence helpers).
  - Prefer small, focused methods (e.g., `createTask`, `updateTask`).

## 5. Navigation & Routes
- Root navigation is grouped by auth state in `RootStack`:
  - `UnAuthenticated`, `Authenticated`, and `Shared` groups.
  - Each screen can include an `if` guard (e.g., `useIsReady`).
- Screen options include modal presentations and sheet behavior when needed.
- A global navigation ref is used for cross-cutting flows (e.g., notifications).
- Home navigation uses a Bottom Tab, with feature drawers/stacks under each tab.
- Navigation is wrapped with `createStaticNavigation` and themed via `withUnistyles`.

## 6. App Providers & Lifecycle
- App root includes:
  - `QueryClientProvider`
  - `I18nextProvider`
  - `SafeAreaProvider`
  - `KeyboardProvider`
- `Portal.Host` for overlays
- `FamilyTaskClientPovider` for session/profile/family state
- App boot flow manages splash screen visibility and notification registration.

## 7. Styling & Theme System
- Styling is handled by `react-native-unistyles` + theme JSON.
- Theme setup (`src/styles/index.ts`) defines light/dark themes, breakpoints, and navigation theme mapping.
- Style creation pattern: `StyleSheet.create((theme, rt) => ({ ... }))`.
  - `theme` provides colors/roundness.
  - `rt` gives runtime info like insets.
- Prefer theme-derived colors and spacing, avoid hard-coded palette values.

## 8. UI Components & Re-exports
- Shared UI elements live in `src/components` and are re-exported via `src/components/index.ts`.
- Feature-only UI can remain inside screen folders.
- Use PascalCase filenames for components and screens.

## 9. Localization (i18n)
- i18next is configured with `task` and `finance` namespaces.
- `useTranslation()` is used inside screens and components.
- `createTranslatableHeaderTitle` wraps React Navigation headers with i18n.
- Language is loaded from persisted preference, with device locale fallback.

## 10. Assets Management
- Assets are mapped in `src/assets/index.ts` (auto-generated by `scripts/genAssets.ts`).
- Access assets via `Assets.get(key)` with typed `AssetKey`.
- Avoid direct `require()` calls outside the asset registry.

## 11. Error & Loading UX
- Use overlay-based feedback (`OverlayView`, `LoadingView`, `ErrorView`).
- Provider-level errors show a full `ErrorLayout` with a retry action.
- Mutations and queries use overlays rather than inline blocking UI.

## 12. Code Style & Types
- TypeScript strict mode; exports should be explicitly typed.
- Keep functions small and predictable; prefer pure helpers.
- Avoid deep relative imports by re-exporting from `index.ts`.
- Formatting uses Prettier: 2 spaces, no semicolons, width 80.

## 13. Rules for AI Agents (Derived)
- Preserve the screen/container pattern (`index.tsx` + `Layout.tsx`).
- Use React Query for all server state; update caches after mutations.
- Never mix UI layout and data fetching in the same component file.
- Add new repositories in `modules/family-task-client` and export via module `index.ts`.
- Keep navigation changes centralized in `src/navigation` and use translation keys for titles.
- Use `react-native-unistyles` for all component styles and theme tokens.
- Add assets through the generator and reference via `AssetKey` only.
- Keep error/loading behavior consistent via shared overlay components.
